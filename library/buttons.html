Integra Core Reference

Button Library

Authoritative reference for agents and templates that render button blocks in WordPress. This page is bound to the live token file and the compiled runtime stylesheet in this plugin, so every supported class combination shown here maps to the real `.in-btn` API and anything not listed here should be treated as unsupported.

Authoring Contract

Use this section as the source of truth when generating WordPress button markup.

Base Formula

<button class="in-btn [size] [variant] [structure]" type="button">
  Label
</button>
<a class="in-btn [size] [variant] [structure]" href="/target">
  Label
</a>

The reference order on this page is base class, size, semantic variant, then structure.

Supported Elements

  • `<button>` Use for form submits and in-page actions.
  • `<a>` Use for navigation and CTA links.
  • Disabled handling `disabled` for buttons, `aria-disabled="true"` for links.

Supported Modifiers

  • Sizes Default is unsuffixed. Optional modifiers: `.in-btn--sm`, `.in-btn--lg`.
  • Variants Primary is the base default. Optional semantic modifiers are `.in-btn--primary`, `.in-btn--secondary`, `.in-btn--success`, `.in-btn--info`, `.in-btn--warning`, `.in-btn--danger`, `.in-btn--light`, `.in-btn--dark`, `.in-btn--white`, `.in-btn--black`.
  • Structures `.in-btn--outline` and `.in-btn--block`.

Do Not Assume

  • No size classes beyond default, `.in-btn--sm`, and `.in-btn--lg`.
  • No visual modifiers for ghost, soft, text, subtle, or pill buttons.
  • No state classes for loading, active, selected, or icon-only buttons.
  • No width modifier besides `.in-btn--block`.

If a block needs any excluded pattern, add it to the component contract first instead of inventing new classes in markup.

Base Size And Structure Surface

The base `.in-btn` renders the default primary button. Size and structure modifiers compose with any semantic variant.

Size Scale

The unsuffixed button is the default size. There is no `.in-btn--md`. 

<button class="in-btn in-btn--sm">Small</button>
<button class="in-btn">Default</button>
<button class="in-btn in-btn--lg">Large</button>

Width Options

<button class="in-btn in-btn--block">Block Button</button>
<button class="in-btn in-btn--block in-btn--lg">Large Block Button</button>

Primary Defaults

`.in-btn` alone is already primary. Adding `.in-btn--primary` is allowed but redundant. 

<button class="in-btn">Primary Filled</button>
<button class="in-btn in-btn--outline">Primary Outline</button>

Composition Examples

<button class="in-btn in-btn--sm in-btn--secondary">Small Secondary</button>
<button class="in-btn in-btn--lg in-btn--outline in-btn--dark">Large Outline Dark</button>
<button class="in-btn in-btn--block in-btn--warning">Block Warning</button>

Rendering Rules

  • Always include `.in-btn`. Nothing renders as a button without the base class.
  • Use at most one size modifier. The valid set is none, `.in-btn--sm`, or `.in-btn--lg`.
  • Use at most one semantic variant. The base button is already primary.
  • Use `.in-btn--primary` only when you need the class to be explicit. It is not required for the default primary render.
  • Add `.in-btn--outline` only when the button should be transparent. Otherwise render the filled version.
  • Add `.in-btn--block` only when the button should span the available width.
  • Inline SVG icons are supported. They inherit `currentColor` and align through the base inline-flex layout.
  • Do not invent extra modifiers in block markup. If a requirement is missing here, it is missing from the component API.
